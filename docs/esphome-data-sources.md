# ESPHome data sources for the CLion plugin

Reference for where ESPHome configuration intelligence comes from, what format
each source is in, and which one this plugin consumes. Researched 2026-06-08
against ESPHome 2026.5.x tooling.

## TL;DR

- ESPHome does **not** publish a JSON Schema. The IntelliJ
  `JsonSchemaFileProvider` route is a dead end.
- Both official tools (the VS Code extension and the new Device Builder
  dashboard) use the same split:
  **schema bundle → completion/hover** · **`esphome config` → validation**.
- There are **three** consumable data products. This plugin consumes the
  **normalized catalog** for completion/hover and **`esphome config`** for
  validation.

## The three data products

### 1. Raw language schema bundle — `schema.esphome.io`

- Generated by `esphome/script/build_language_schema.py` from ESPHome's
  Voluptuous (`CONFIG_SCHEMA`) validators; docs injected afterward by
  `esphome.io/script/schema_doc.py`.
- Published per ESPHome release by the `esphome/esphome-schema` GitHub Action
  to Cloudflare R2: `https://schema.esphome.io/{version}/schema.zip`
  (e.g. `2026.4.3`, plus `dev` and beta channels). Also attached to
  `esphome-schema` GitHub releases.
- The zip holds one JSON file per component plus a `core` index.
- **Custom format, NOT JSON Schema.** Top level is `core` →
  `{ components, platforms, action, condition }`. A config option looks like:

  ```json
  "delay": {
    "type": "schema",
    "templatable": true,
    "schema": { "extends": ["core.positive_time_period_milliseconds"] },
    "docs": "This action delays execution..."
  }
  ```

  Vocabulary: `type` (`schema`/`registry`/`integer`/…), `config_vars`,
  `schema.extends` (inheritance — resolve transitively), `registry`
  (polymorphic references: filters, actions, effects), `templatable`
  (accepts `!lambda`), `is_list`, `key` (`Required`/`Optional`), `docs`.
  No `$schema`/`properties`/`$ref`.
- The hard parts (platform discriminators, `extends` chains, registry
  expansion, pin schemas) live here **unresolved** — using this directly means
  reimplementing ESPHome's resolution logic.

### 2. Normalized catalog — Device Builder `definitions/` (**what we consume**)

- Produced by `device-builder/script/sync_components.py`: downloads the raw
  `schema.zip`, resolves `extends`/registries, augments with light live
  `esphome` introspection (`cv.SplitDefault` → `platform_defaults`,
  `multi_conf`, `supported_platforms`, `cv.only_on`, `cv.has_*_one_key` →
  `required_groups`), scrapes image URLs from the docs `index.mdx`, and emits
  **baked, normalized JSON**.
- Apache-2.0 code; output is plain JSON, regenerated every ESPHome release.

  | File | Contents |
  |---|---|
  | `components.index.json` | Slim index, tagged `esphome_schema_version`. One entry per **resolved platform** — id is `<domain>.<platform>` (`sensor.dht`, `switch.gpio`). |
  | `components/<id>.json` | Per-component body: full `config_entries` tree. |
  | `automations.index.json` + `automations/` | Actions/conditions/triggers. |
  | `boards.index.json` + `board_bodies/` | Board catalog with per-GPIO pin maps. |
  | `pin_registry_modes.index.json` | Pin mode registry. |

- The **`sensor:`/`platform:` discriminator is already resolved** — each
  platform is its own catalog id with its own `config_entries`. That is exactly
  the lookup a text-completion engine needs at `sensor: - platform: dht`.
- The `ConfigEntry` model (source: `device-builder/.../models/common.py`)
  flattens every field into typed, declarative attributes:
  - `type` ∈ `{string, secure_string, integer, float, boolean, pin,
    time_period, float_with_unit, icon, id, trigger, color, mac_address,
    lambda, json, nested, map, registry_list, label, divider, alert, unknown}`
  - constraints/metadata: `required`, `default_value`, `platform_defaults`,
    `supported_platforms`, `options` + `allow_custom_value`, `range`,
    `display_format` (`hex`), `unit_options`, `pin_features`/`pin_mode`,
    `registry`, `required_groups`
    (`exactly_one`/`at_least_one`/`at_most_one`/`none_or_all`), `templatable`,
    `multi_value`, `depends_on`(`_value`/`_not`/`_component`),
    `references_component`, `advanced`, `hidden`, `help_link`, `group`,
    `config_entries` (recursive), `platform_type`.

### 3. The running tool — `esphome config` (**what we validate with**)

- Ground-truth validation. Device Builder uses `devices/validate {configuration}`
  (streaming, spawns `esphome`); the VS Code extension uses a WebSocket to the
  dashboard at `ws://<host>:6052/vscode`. CLI equivalent: `esphome config <file>`
  (non-zero exit ⇒ invalid).
- Required for correctness: it resolves `!secret`/`!include`/`!lambda`,
  substitutions, packages, and platform conditionals that no static schema
  captures.

## Decision for this plugin

| Concern | Source | Mechanism |
|---|---|---|
| Completion / hover / docs | Normalized catalog (#2) | Vendor a copy pinned to an ESPHome version; refresh per release |
| Validation / diagnostics | `esphome config` (#3) | `ExternalAnnotator` shelling out to the user's `esphome` |
| Fallback / cross-check | Raw bundle (#1) | Only if the normalized catalog is unavailable |

We consume the normalized catalog rather than the raw bundle so we do **not**
reimplement ESPHome's schema-resolution logic — `sync_components.py` already
does it and is maintained per release.

### Due-diligence items

1. **Licensing — resolved (Apache-2.0).** The baked `definitions/` JSON are
   git-tracked source files in the `device-builder` repo, whose `LICENSE` is
   Apache-2.0 (`pyproject.toml` also declares `license = "Apache-2.0"`).
   `esphome`, `esphome.io`, `esphome-schema` and `device-builder` are all under
   the same ESPHome GitHub org, so the upstream rights holder has released the
   aggregated, generated output under Apache-2.0 — no cross-license concern for
   a downstream consumer. To vendor the files in the plugin:
   - bundle the Apache-2.0 license text (e.g. `third_party/esphome-device-builder/LICENSE`);
   - add a one-line attribution to `esphome/device-builder`;
   - no `NOTICE` obligation (upstream ships no `NOTICE` file);
   - state changes only if the JSON is modified (it won't be — vendored as-is).
2. **Contract stability — mitigated by pinning.** `definitions/` is Device
   Builder's internal format with no compatibility guarantee, so pin to a
   tagged `device-builder` release / fixed commit and upgrade deliberately per
   ESPHome release. Belt-and-suspenders: the index self-identifies its
   `esphome_schema_version` (detect mismatches) and the decoder is tolerant
   (`ignoreUnknownKeys` + `coerceInputValues`, so drift degrades gracefully).
   Keep the raw-bundle path only as a deep fallback.
3. **`pin_features` evolution** — decoded as `List<String>` (not an enum) so an
   unknown feature string from a future release parses cleanly.

## Key source pointers

- `esphome/script/build_language_schema.py` — raw bundle generator.
- `esphome/esphome-schema` `.github/workflows/generate-schemas.yml` — publish
  pipeline (R2 upload, versioned paths).
- `esphome/device-builder` `script/sync_components.py` — catalog generator
  (the normalization we rely on).
- `esphome/device-builder` `esphome_device_builder/models/common.py` &
  `models/components.py` — the `ConfigEntry` / `ComponentCatalogEntry` models
  mirrored by `prototype/`.
- `esphome/device-builder` `docs/API.md` — `devices/validate` streaming command.
